# Mouse-only software KVM architecture across Windows and macOS

## What this is called

What you are building is usually called a **software KVM**, **mouse sharing**, **cross-computer cursor sharing**, or, in the research literature, **pointer redirection**. Open-source projects such as Input Leap, Barrier, Deskflow, Lan Mouse, and Cursr all explicitly describe themselves as software KVMs or software for sharing a mouse across multiple computers. The Stanford **PointRight** paper uses the more precise academic term **pointer redirection**, which is useful because you are not doing video capture, remote desktop, or hardware KVM switching; you are transferring control of the pointer between independent desktops running on different machines and operating systems. citeturn8view1turn8view2turn8view0turn8view3turn32view0turn15view1

For your scope, "KVM" is slightly broader than you need, because most shipping products bundle keyboard, clipboard, and sometimes file transfer. But the mouse path is separable. The Barrier protocol defines mouse-enter, mouse-leave, mouse-move, relative-move, button, and wheel messages independently of its keyboard messages, and practical remote-control implementations on macOS likewise construct mouse events separately from keyboard events. That means a **mouse-only KVM** is a perfectly coherent design target rather than an odd corner case. citeturn10view0turn23view0

## How existing systems handle mouse sharing

Today's mouse-sharing systems fall into two broad families. The first is the classic **LAN software KVM** model used by Deskflow, Input Leap, Barrier, Lan Mouse, Cursr, Logitech Flow, and Mouse Without Borders: each machine keeps rendering its own desktop, but one machine's mouse is allowed to cross a configured boundary and take control of another machine. Deskflow describes itself as a software KVM with TLS enabled by default; Input Leap and Barrier use the same basic edge-crossing model; Lan Mouse emphasises performance and says it encrypts network traffic with DTLS; Cursr focuses on configurable **border links** and segmented links between display edges; Logitech Flow and Apple Universal Control both present the user-facing metaphor of "move the pointer to the edge and continue onto the next computer." citeturn8view0turn8view1turn8view2turn8view3turn32view0turn32view1turn32view2

The second family is **remote control**, where one machine actively injects events into another but does not necessarily try to preserve a seamless multi-display illusion. Multi's engineering write-up is a good modern example from that world: it posts synthetic `CGEvent`s on macOS, keeps latency low, and adds a "short-circuit" mechanism so the local user can immediately reclaim control if both parties start acting at once. Even though that article is about collaborative remote control rather than a software KVM, the conflict-resolution ideas are directly relevant to your build. citeturn23view0

Among the open systems, Barrier-like protocols are particularly instructive because their state machine is explicit. The protocol includes a handshake, screen information exchange, **enter** and **leave** messages, and separate **absolute** and **relative** mouse-movement messages. Deskflow's current protocol definitions also expose a default TCP port of `24800`, a 3-second keep-alive rate, and a protocol design that distinguishes a controlling **primary** side from a receiving **secondary** side. That is a very useful blueprint even if you ultimately do your own UDP-based design, because it cleanly separates topology, ownership transfer, and motion delivery. citeturn10view0turn12view0

## How the pointer transitions from one screen to another

The cleanest mental model is not "one cursor travelling between computers," because that is not what the operating systems expose. The real model is **ownership transfer**. While the pointer is on machine A, machine A owns and displays its native cursor. When the user pushes through a configured boundary, machine A stops being the active pointer target, emits a **leave** event, and machine B receives an **enter** event plus an initial coordinate. After that, machine B owns and displays its own native cursor. PointRight described this as a geometric model for redirecting input across independent machines, and Barrier-style protocols encode it explicitly as `COUT`/leave and `CINN`/enter messages. citeturn15view1turn10view0

This transition logic is normally driven by a **topology model**. Apple's Universal Control tells users to arrange device displays relative to one another, then physically "push through" the left or right edge in the general direction of the other device; once the pointer begins to "push through," continuing the gesture completes the connection. Cursr exposes the same idea in a more developer-looking form by letting users create **virtual links between borders** and even split borders into **resizable segments**. The research literature arrives at the same conclusion from the opposite direction: PointRight uses a geometric room/display model, and later work on automatic cross-display navigation generates spatially consistent transitions from a calibrated model of display relationships. citeturn32view2turn32view0turn15view1turn18search1

The difficult part is not the crossing itself; it is **preserving spatial continuity** when the displays differ in size, resolution, offset, or scaling. Baudisch et al.'s **Mouse Ether** paper showed that naïve operating-system pointer motion causes visible "warping" when displays have different resolutions or offsets, and that transforming mouse movement to match the geometry users actually see improved target acquisition by up to 28% in their study. Benko and Feiner's **Multi-Monitor Mouse** then explored deliberate pointer warping between screens and found strong user preference and speed gains, while their later **Pointer Warping in Heterogeneous Multi-Monitor Environments** paper reported improvements of up to about 30% in some cases when using a frame-memory warping strategy in heterogeneous setups. These papers matter for you because a Windows-to-macOS handoff has exactly the same geometry problem, just across a network instead of inside one workstation. citeturn15view0turn15view2turn15view4

A practical route model for a mouse-only software KVM is therefore:

1. represent each local display as a rectangle in a canonical space;
2. represent each allowed transition as a directed edge link;
3. store the **fractional position** along the exit edge, not just raw pixels;
4. on crossing, map that fraction to the destination edge or edge segment;
5. enter the destination with an absolute coordinate once, then continue with relative deltas most of the time.

That synthesis is exactly what modern products and the academic literature converge on: **absolute positioning for handoff and recovery, relative motion for continuous movement, and a topology graph for routing**. citeturn10view0turn32view0turn15view1turn15view0turn15view4

## How to stop local cursor conflict and make only one cursor appear active

The central implementation problem is **not** moving a "shared" cursor. It is preventing the source computer's local pointer pipeline from fighting with the destination computer's injected pointer pipeline. Multi's macOS remote-control write-up describes the failure mode exactly: if both local and remote input act at once, the cursor can "fight" between disparate positions and visibly jitter. Their solution is to tag synthetic remote events and let truly local events short-circuit the session. That is the right general pattern for a software KVM too. citeturn23view0

On **Windows**, Microsoft's guidance strongly points to **Raw Input** for capture. Raw Input gives you device-level mouse data through `WM_INPUT`, can work in the background with `RIDEV_INPUTSINK`, and supports batched reads via `GetRawInputBuffer`, which is specifically useful for high-frequency mice such as 1000 Hz devices. Microsoft also shows `RIDEV_NOLEGACY` for the case where you want raw mouse input while suppressing legacy mouse window messages to your application window. For precision, Microsoft's "high-definition mouse movement" guidance says `WM_INPUT` reflects the raw HID results, while `WM_MOUSEMOVE` is screen-resolution-limited and has ballistics applied. In other words: **capture with raw relative deltas if you care about feel, not with pixel-position messages**. citeturn34view0turn34view1turn35search0turn36view0

If you need to **block** local pointer motion on Windows while the remote machine is active, the tool most people reach for is `WH_MOUSE_LL`. Microsoft documents that a low-level mouse hook can return a nonzero value to prevent the system passing the message further. But Microsoft also says these hooks must run very quickly, are silently removed on timeout, and that in most cases raw input is a better monitoring mechanism. That leads to a good division of labour: use Raw Input for high-rate capture, and use a low-level hook only as a narrow control gate when you must suppress the local stream during remote ownership. citeturn37view0

For **event injection** on Windows, use `SendInput`. Microsoft states that `SendInput` inserts events serially into the mouse/keyboard input stream, and that injected mouse events can carry application-defined metadata in `MOUSEINPUT.dwExtraInfo`. At the receiving side, `MSLLHOOKSTRUCT` exposes both `dwExtraInfo` and the injected-event flags `LLMHF_INJECTED` and `LLMHF_LOWER_IL_INJECTED`, which means you can reliably distinguish your own synthetic packets from genuine hardware-originated motion. This is the Windows equivalent of tagging remote events on macOS. Also note that `SendInput` is subject to **UIPI**, meaning you can only inject into apps at equal or lower integrity level. citeturn24search4turn24search0turn25search0

Windows also gives you two useful cursor-control primitives. `ClipCursor` confines the pointer to a rectangle, which is handy if you want to "park" the local cursor into a 1×1 region while the remote machine owns the session. `ShowCursor` changes visibility, but it operates through a display counter rather than a simple boolean, so it is useful as a visibility aid rather than a full state-management strategy. For absolute positioning you can use `MOUSEEVENTF_ABSOLUTE`; if you also set `MOUSEEVENTF_VIRTUALDESKTOP`, the coordinates map to the entire virtual desktop rather than just the primary monitor. citeturn3search17turn24search0turn19search5turn19search21

On **macOS**, the corresponding stack is **Quartz Event Services**. Apple describes event taps as filters for observing and altering the stream of low-level user input events in macOS. `CGEventTapCreate` creates the tap, `CGEvent.post` posts a synthetic event, and `CGEventField.eventSourceUserData` gives you a 64-bit user-supplied field you can use to mark remote events. That is exactly why Multi uses it: events carrying the tag are recognised as synthetic remote events, while untagged events are treated as truly local and allowed to reclaim control. citeturn39search0turn4search6turn31search0turn23view0

Apple also provides explicit cursor primitives. `CGDisplayHideCursor` hides the cursor and increments the hide count; `CGAssociateMouseAndMouseCursorPosition` connects or disconnects the mouse and cursor while an application is in the foreground; `CGWarpMouseCursorPosition` moves the cursor without generating or posting an event; and `CGDisplayMoveCursorToPoint` moves the cursor relative to a chosen display. Those APIs are enough to implement the normal software-KVM handoff pattern on macOS: hide or park the local cursor on the source machine and inject/display the native cursor on the destination machine. ## Networking, latency, and jitter control

For mouse motion, the fastest design is almost never "reliably deliver every move packet." Mouse movement is **state**, not a log that must be replayed losslessly. If packet 120 is an older position and packet 121 is the newest position, retransmitting 120 after 121 usually hurts more than it helps. That is why the most promising transports for this problem are **hybrid**: a lossy channel for motion, and a reliable ordered channel for control-plane events such as connect, topology, enter, leave, and occasional absolute resyncs. QUIC is a strong match because RFC 9000 gives you a secure transport over UDP, RFC 9221 adds unreliable datagrams, and the reliable stream path remains available for state that must not be dropped. WebRTC data channels are another fit, because RFC 8831 requires support for both reliable and unreliable channels and requires congestion control. citeturn28view1turn28view2turn28view3

If you do keep your own UDP protocol, follow the transport guidance rather than treating the LAN as "free." RFC 8085 emphasises that UDP has **no inherent congestion control**, recommends using established congestion-control techniques, and warns that IP fragmentation reduces efficiency and reliability because losing a single fragment loses the whole packet. The same RFC advises that if you fragment application messages yourself, each datagram should still be independently useful and independently retransmittable. QUIC's own transport specification goes further and says QUIC UDP datagrams **must not be fragmented at the IP layer**, with PMTU discovery used instead. For a mouse protocol, this all pushes toward **small packets, no fragmentation, and no retransmission of stale motion**. citeturn26search4turn28view0turn29view0turn29view1turn29view2

Lan Mouse is notable because its authors arrived at a similar practical conclusion from the application side: they focus on performance and encrypt all network traffic with **DTLS**. That makes it a useful precedent for your current UDP approach. If you want to stay custom, **DTLS plus a split reliable/unreliable application protocol** is a defensible design; if you would rather stop rebuilding transport machinery, **QUIC with reliable streams plus DATAGRAM** is the cleaner modern option. citeturn8view3turn28view2turn28view1

On the human-factors side, the pointing literature is very clear that both latency and jitter matter, but **mean latency is usually the bigger villain**. Recent work reported that increases in mean latency caused the strongest impairments in target acquisition speed and accuracy, while large jitter amplitudes still hurt but the frequency of jitter changes did not show a significant effect in that study. Earlier work likewise found measurable harm from added latency jitter and dropouts in pointing tasks. The practical implication is that your first optimisation target should be **lowering and stabilising end-to-end latency**, not building an elaborate smoothing filter that adds delay. citeturn17search3turn17search1turn17search5turn14search1

That leads to a concrete performance policy for mouse motion:

- capture **relative** deltas at the source with high-frequency support;
- timestamp and sequence every motion packet;
- send only the freshest motion state on the lossy path;
- coalesce bursty motion rather than queueing it;
- use occasional **absolute** correction packets to bound drift;
- keep any jitter buffer tiny or zero on a LAN;
- keep hooks and event taps as thin as possible, because both platforms penalise slow handlers. citeturn34view0turn36view0turn10view0turn37view0turn23view0

## A practical architecture for your build

If I were building a **mouse-only Windows↔macOS software KVM** from scratch today, I would structure it as five layers.

The **topology layer** would model every display as a rectangle in canonical logical coordinates and every legal crossing as a directed route between edge segments. Fractional edge position would be preserved on handoff so that a crossing at 37% down the right edge of one display enters at 37% down the mapped edge segment on the destination. That is the simplest route model that still supports Apple-style arrangement, PointRight-style geometry, and Cursr-style segmented border links. citeturn32view2turn15view1turn32view0

The **ownership layer** would implement a strict state machine: `LocalActive`, `CrossingPending`, `RemoteActive`, `ReturnPending`, and `Resync`. A crossing would emit one reliable **leave**, one reliable **enter**, and one absolute coordinate on the destination. After that, only relative deltas would flow until drift, display change, packet loss, or a route change forced an absolute resync. Barrier-style protocols already prove that this split works in practice. citeturn10view0turn12view0

The **capture/injection layer** would be OS-specific. On Windows I would capture with Raw Input, optionally suppress with a very thin `WH_MOUSE_LL` gate during remote ownership, and inject with `SendInput`, tagging every synthetic event in `dwExtraInfo` and ignoring injected events when monitoring. On macOS I would capture and filter through a CGEvent tap, tag synthetic events with `eventSourceUserData`, inject with posted `CGEvent`s, and hide or disassociate the local cursor during remote ownership if the app is foreground. On both platforms, I would treat "local user touched the mouse while remote active" as an immediate reclaim unless the event is tagged as synthetic. citeturn34view0turn37view0turn24search4turn25search0turn39search0turn31search0turn40search0turn40search4turn23view0

The **transport layer** would use two paths. Motion would go over an unreliable unordered path; control would go over a reliable ordered path. If you want to stay close to your current UDP work, DTLS plus an application-level split is reasonable. If you want the transport to do more of the heavy lifting, QUIC streams plus QUIC DATAGRAM is a stronger foundation and aligns with current standards. Either way, keep packets small and avoid fragmentation. citeturn8view3turn26search4turn28view1turn28view2turn29view0turn29view1

The **recovery layer** would watch for display changes, permissions failures, missed keep-alives, and route mismatches. When the receive side changes resolution or layout, the Barrier protocol's pattern is instructive: the client reports new screen information and ignores movement until it gets an acknowledgement, preventing off-screen movement into stale geometry. That exact idea is worth copying. citeturn10view0
